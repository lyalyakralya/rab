# ПР №10. Лицензирование и open-source compliance

## 1. Выбор лицензии

**Выбранная лицензия:** MIT
**Тип:** разрешительная (permissive)

**Что разрешает:**
- Использование кода в любых целях
- Копирование и распространение
- Модификацию и создание производных работ
- Коммерческое использование
- Сублицензирование

**Что требует от пользователей кода:**
- Сохранять уведомление об авторских правах
- Сохранять текст лицензии

**Может ли кто-то продавать ваш код:** Да, может. MIT позволяет продавать код как часть коммерческого продукта без раскрытия исходников.

**Откуда GitHub знает какая у репозитория лицензия:**
GitHub использует библиотеку licensee для сканирования файлов LICENSE, COPYING или COPYRIGHT в корне репозитория. Она сравнивает содержимое с базой известных лицензий и определяет совпадение.

## 6. Анализ совместимости

**Сценарий 1 (MIT + GPL в коммерческом):**
Нет, нельзя. GPL требует раскрытия исходного кода всего продукта. Если включить GPL-библиотеку в закрытый коммерческий продукт, он должен стать открытым под GPL.

**Сценарий 2 (Apache-2.0 + MIT):**
Да, можно. MIT совместима с Apache-2.0. MIT не требует ничего кроме сохранения уведомления, что не конфликтует с Apache.

**Сценарий 3 (MIT без указания авторства):**
Да, нарушает. MIT требует сохранения уведомления об авторских правах и текста лицензии. Без указания авторства использование противоречит условиям.

**Сценарий 4 (AGPL в SaaS):**
Нет, нельзя. AGPL требует открытия исходного кода даже при использовании через сеть (SaaS). Это ключевое отличие от GPL.

**Сценарий 5 (нет LICENSE):**
Нет, нельзя. Без лицензии действует авторское право по умолчанию - код можно смотреть, но нельзя использовать, копировать или распространять.

Ответы на контрольные вопросы
1. Чем разрешительная лицензия (MIT) отличается от копилефтной (GPL)?
MIT - разрешает всё, требует только сохранить уведомление об авторстве
GPL - требует, чтобы производные работы тоже были под GPL (вирусный эффект)

2. Что такое SPDX-идентификатор?
Стандартный короткий идентификатор лицензии (например MIT, Apache-2.0). Машиночитаемая форма нужна для автоматической проверки лицензий и составления SBOM.

3. Компания взяла GPL-2.0 код в закрытый продукт
Нарушает условие распространения исходного кода. Последствия: судебные иски, обязательство открыть исходники, финансовые компенсации.

4. Что такое SBOM?
Software Bill of Materials - список всех компонентов. Нужен для compliance (знание лицензий) и безопасности (быстрая проверка уязвимостей).

5. Репозиторий без LICENSE - можно ли использовать?
Нельзя. Без лицензии действует авторское право по умолчанию - "все права защищены".

6. Чем AGPL отличается от GPL?
AGPL закрывает "облачную лазейку" - требует открытия исходников даже при использовании через сеть (SaaS), а не только при распространении.

7. Зачем проверять лицензии в CI?
Зависимости обновляются и добавляются постоянно. CI проверяет каждый коммит, не позволяя случайно добавить несовместимую лицензию.

## Выводы

В ходе практической работы были изучены:

1. **Лицензирование** - важный юридический аспект разработки ПО. Без лицензии код нельзя использовать.

2. **SPDX** - стандарт для машиночитаемого описания лицензий. Позволяет автоматизировать проверку.

3. **REUSE** - спецификация для единообразного указания лицензий в каждом файле.

4. **Инструменты compliance** - licensee, reuse, pip-licenses, trivy помогают автоматически проверять лицензии.

5. **CI-проверки** - позволяют автоматически контролировать лицензии в процессе разработки.

**Самое важное открытие:** нельзя просто взять код с GitHub без лицензии. Это нарушение авторских прав, даже если код доступен для просмотра.
