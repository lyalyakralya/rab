ПР №3. Права доступа Linux и управление пользователями

1. Пользователи и группы

Созданные пользователи и их роли:

| Пользователь | Группа      | Роль                    |
|--------------|-------------|-------------------------|
| bob          | developers  | Администратор проекта   |
| karl         | developers  | Разработчик             |
| masha        | auditors    | Аудитор                 |

Поля /etc/passwd

Строка для bob: bob:x:1001:1001::/home/bob:/bin/bash

Разбор полей:
1. bob - имя пользователя
2. x - признак, что пароль хранится в /etc/shadow
3. 1001 - UID (идентификатор пользователя)
4. 1001 - GID (идентификатор основной группы)
5. пусто - комментарий (GECOS)
6. /home/bob - домашняя директория
7. /bin/bash - командная оболочка

В /etc/shadow хранятся хэши паролей и политики их действия. Файл недоступен обычным пользователям для предотвращения взлома хэшей и получения доступа к системе.

2. Права доступа chmod/chown

- `/srv/project/code`: права 770 (символьная: rwxrwx---)
- Владелец: bob:developers

Проверка доступа
- karl (в группе developers): При правах 750 (rwxr-x---) karl НЕ МОГ создать файл, потому что группе было разрешено только чтение и выполнение (r-x), но не запись (w)
- masha (не в группе): masha НЕ МОЖЕТ войти в директорию, потому что права для остальных (other) --- --- (нет доступа)


Файл q1.txt
Права: 640 (символьная: rw-r-----)
- Владелец (bob): чтение + запись
- Группа (auditors): только чтение
- Остальные: нет доступа

3. ACL

Задача: дать masha доступ на чтение без изменения группы.

Команда: `sudo setfacl -m u:masha:r-x /srv/project/code`
 Вывод getfacl после изменения:
file: /srv/project/code
owner: bob
group: developers
user::rwx
user:masha:r-x
group::rwx
mask::r-x
other::---

Преимущества ACL
ACL позволяют дать доступ конкретному пользователю (masha) без:
- Изменения группы владельца файла
- Добавления masha в группу developers
- Нарушения существующей модели безопасности
 4. sudo-политики

| Пользователь | Разрешено                                    | Ограничение                    |
|--------------|----------------------------------------------|--------------------------------|
| bob          | Всё (NOPASSWD)                               | Нет                            |
| karl         | apt, apt-get                                 | Только пакетный менеджер       |
| masha        | journalctl, cat /var/log/*                   | Только просмотр логов          |


 Принцип безопасности
- bob использует NOPASSWD как администратор для автоматизации
- karl не использует NOPASSWD для apt, чтобы требовать подтверждение перед критическими действиями
- Реализован принцип наименьших привилегий (Least Privilege)

5. PAM

 Модуль pam_unix.so
В /etc/pam.d/sudo строка: `auth required pam_unix.so`

required означает, что модуль ДОЛЖЕН успешно выполнить проверку. При неудаче аутентификация немедленно прекращается с ошибкой.

Политика паролей в pam_pwquality
Текущие требования (из /etc/pam.d/common-password):
- `minlen=8` (минимальная длина 8 символов)
- `difok=3` (минимум 3 символа должны отличаться от старого пароля)
- `reject_username` (пароль не должен содержать имя пользователя)
