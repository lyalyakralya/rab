ПР №4. Аудит событий: journalctl и auditd

 1. journalctl — системный журнал

 Хост: debian
 Сервис: sudo — команда с повышенными привилегиями
 Пользователь: katya — кто выполнил sudo

мая 05 21:51:41 debian kernel: Bluetooth: hci0: unexpected cc 0x0c12>

Статистика входов:

 Успешных входов: 0 (вставить число из journalctl _COMM=sshd | grep "Accepted" | wc -l)
 Неудачных попыток: 0 (вставить число из journalctl _COMM=sshd | grep "Failed" | wc -l)

2. auditd — настройка правил

Применённые правила (вывод auditctl -l)
-w /etc/passwd -p rwa -k auth-files
-w /etc/shadow -p rwa -k auth-files
-w /etc/group -p rwa -k auth-files
-w /etc/sudoers -p rwa -k priv-files
-w /etc/sudoers.d -p rwa -k priv-files
-w /etc/ssh/sshd_config -p rwa -k ssh-config
-a always,exit -F arch=b64 -S open,openat -F success=0 -F key=access-denied
-a always,exit -F arch=b64 -S unlink,unlinkat -F key=file-delete
-a always,exit -F arch=b64 -S execve -F euid=0 -F auid!=0 -F auid!=-1 -F key=priv-exec
Разбор записи SYSCALL

| Поле | Значение | Что означает |
|------|----------|---------------|
| auid |   1000   | Логический UID пользователя (оригинальный, не меняется при sudo) |
| uid  |     0    | Реальный UID процесса (0 = root после sudo) |
| euid |     0    | Эффективный UID (используется для проверки прав доступа) |
| comm |   "cat"  | Имя команды (исполняемого файла) |
| exe  | "/usr/bin/cat" | Полный путь к исполняемому файлу |

3. Расследование инцидента

Сценарий:Пользователь karl совершал подозрительные действия. Специалист по ИБ (katya) проводит расследование.

 Хронология действий karl:

Действие | Результат | Источник (команда поиска) |
|----------|-----------|---------------------------|
| cat /etc/shadow | Отказ  | ausearch -k access-denied -i |
| ls /root | Отказ  | ausearch -k access-denied -i |
| cat /etc/passwd | Успех (разрешено) | ausearch -k auth-files -i |
| touch /tmp/totally-not-malware.sh | Успех | grep "totally-not" /var/log/audit/audit.log |
| echo '#!/bin/bash' > /tmp/totally-not-malware.sh | Успех | grep "totally-not" |
| cat /etc/sudoers | Отказ (Permission denied) | ausearch -k access-denied -i |

Как auditd помог расследованию:

1. Зафиксировал все неудачные попытки доступа к защищённым файлам
2. Сохранил auid нарушителя (karl), даже если бы он использовал sudo
3. Позволил найти создание подозрительного файла через системные вызовы
4. Предоставил точную временную метку каждого события

 Чем auditd лучше journalctl для задач аудита безопасности:

| Критерий | auditd | journalctl |
|----------|--------|------------|
| Уровень работы | Ядро (системные вызовы) | Пользовательский (логи сервисов) |
| Что фиксирует | Все системные вызовы (open, read, write, execve) | Только то, что логирует приложение |
| Обход злоумышленником | Почти невозможен — на уровне ядра | Возможен — очистка логов, подмена сервисов |
| Детализация | Поля: auid, uid, euid, comm, exe, syscall | Только сообщение от сервиса |

 5. Контрольные вопросы

1. Чем принципиально отличается auditd от journald? Почему нельзя обойтись только journald?

 journald логирует то, что приложения сами пишут в журнал
 auditd работает на уровне ядра и фиксирует системные вызовы независимо от приложений
 Нельзя обойтись только journald, т.к. злоумышленник с правами root может очистить журналы или отключить логирование. auditd это предотвратить сложнее.

2. Что такое auid и почему оно важнее uid при расследовании инцидентов с sudo?

 auid (Audit User ID) — назначается при логине и не меняется при использовании sudo
 uid может меняться (например, после sudo становится 0/root)
 auid позволяет узнать, кто именно выполнял команду, даже через sudo

3. Что означает флаг -k в правиле auditd? Для чего он используется?

 -k добавляет поисковый ключ (метку) к событию
 Используется для фильтрации в ausearch (например, ausearch -k access-denied)

4. Чем -w (watch) отличается от -a always,exit в правилах auditd?

 -w — следит за файлом или директорией (проще в использовании)
 -a always,exit — добавляет правило на системный вызов с фильтрацией (гибче)

5. Злоумышленник удалил файл /var/log/audit/audit.log. Как это помогает ему скрыть следы и как от этого защититься?

 Помогает скрыть следы, т.к. удаляются записи о его действиях
 Защита: настроить правило на unlink, отправлять логи на удалённый сервер, установить неизменяемые атрибуты (chattr +a)

6. Что нужно сделать чтобы правила auditd сохранились после перезагрузки?

 Поместить правила в файл /etc/audit/rules.d/*.rules
 Использовать augenrules --load для применения
